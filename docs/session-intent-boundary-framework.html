Product Requirements Document

Session Intent Boundary Intelligence

把 session 从「长聊天记录」升级为「可维护的任务容器」：在一句真正属于另一个任务的输入混入、造成不可逆的会话污染之前，用模型判断边界，并自动组织 fork、handoff 或 compact context。

防止不可逆污染 真切换混入后，session 注意力被带偏，无法复用，只能整条抛弃。
必须使用模型 任务边界是语义判断，不是关键词规则。
原生感的兑现 不是变现点，是 Zo「帮你管理工作流」心智的一次具体落地。

01背景与问题

在一个 session 里推进任务 A 时，用户常会被一个真正无关的任务 B 打断——「突然发现一个 bug」「对了顺便算一下另一个功能的 ROI」。如果顺手在当前 session 里问了 B，问题虽然解决了，但会话被搞脏了：模型注意力被 B 带偏，A 的上下文被稀释。

核心问题是「不可逆」。一旦真切换的内容进入 session，这条 session 的纯度就回不来了——compact 也救不回，因为污染已经进了历史、影响后续每一轮的注意力分布。用户最终只能把整条 session 抛弃，重新开一条、再手动把 A 的上下文带过去。这是一次真实发生的损失，不是一个理论上的洁癖。
用户研究证据

大需求方向已被验证

用户研究（n=2，ZooAI/user-research）印证「上下文 / 记忆的可寻址性」是 2/2 出现的真需求：用户在手动复制上下文、手搓 CLAUDE.md 来绕过。本能力是这一需求的一个特定子问题——让属于另一个任务的输入不要污染当前主线。

证据等级要诚实

具体切口仍是强假设

「分流主线、防止不可逆污染」这一具体切口目前主要来自创始人自身真实体感，属待证伪假设。同一研究还有两条硬约束：用户「为模型付费、为体验忍耐」，痛点「能忍、非 hair-on-fire」——意味着用户不会主动呼唤它，对打断的耐受度更低。

02产品定位

这个能力不是「偏题检测」。偏题检测把系统摆在纠正用户的位置，容易冒犯，也容易退化成烦人的确认弹窗。它真正防的不是「跑题」，而是一次不可逆的会话损坏。这一点决定了它的两条设计约束：

约束 A · 时机

必须在发送前拦截

因为损坏不可逆，正确的介入时机是消息进入 session 之前那一瞬，而不是事后提醒——消息一旦发出、被模型看到，损坏已经发生，再弹窗毫无意义。pending message 先进判别器，再决定是否进主 session。

约束 B · 误判不对称

漏拦远比误拦贵

漏拦一次（false negative），用户损失一整条 session；误拦一次（false positive）在「发送前」这个时机下只是多问一句「要不要单独开」，用户一秒可消除，且什么都没被损坏。生死线不在「要不要拦」，而在判别器能否分清下面两件事。

推荐命名：Session Intent Boundary Intelligence。它属于 Session OS 的一层能力，负责在不可逆污染发生前识别任务边界、保护主线上下文、生成跨 session 交接，并把复杂的 context engineering 从用户手里拿走。
它是护城河拼图，不是变现点。Zo 是开源 Agent、本身就是编排层，模型能力依赖外部，主力用户是非研发的日常 / 创作 / PM / 设计师。这类用户不会自己做 context engineering（研发才会写 CLAUDE.md、手动 fork），他们只会模糊地觉得「这个 AI 后来怎么不好用了」然后弃掉、甚至怪到模型头上。Zo 替他们在那一瞬挡掉，正是「原生感」的具体兑现——它和自动命名、并行面板、自动 handoff 是同一类能力：把研发用户手动做的 context 管理，替非研发用户默默做掉。它本质是已有能力（Compact / Branch / Session 创建）之上的产品层编排，不是第六个新功能，迁移成本近乎零。

03目标与非目标

Goals · 目标
  • 在「真切换」输入进入 session 前识别它，避免不可逆污染。
  • 当用户接受分流时，自动生成高质量 handoff，让 fork 后比留在原会话更省事。
  • 后台持续维护一份可追溯的 Session Task Brief，作为边界判断与交接的事实来源。
  • 复用已有 Compact / Branch / Session 创建能力，以产品层编排呈现，而非新增独立功能。
  • 把每一次系统介入做成「惊喜」而非「负担」——用户的反应应是「它怎么知道」，不是「它又来问我了」。
Non-Goals · 非目标
  • 不做「偏题检测 / 纠正用户」式的体验，文案不审判用户。
  • 不靠关键词、相似度等启发式冒充语义判断。
  • 不在事后（消息已发送后）弹窗提醒——那既无法阻止污染，又最伤信任。
  • 不把本能力当作独立的付费 / 变现卖点来承诺商业指标。
  • 不拦截「自然漂移」（主线内的追问、确认、试错），不制造高频打断。
  • 不判别 / 推荐 branch（回到历史节点探索替代路线）：判别器只看一句前向的待发送输入，无法可靠反推用户想回到哪个历史节点，强行纳入会扩大判别空间、拉低准确度。branch 仍是用户主动触发的已有机制，不在本能力范围内。
  • 第一阶段不追求覆盖全部动作分支，先打通最小闭环（见路线图）。

04目标用户与典型场景

主力用户是非研发的日常 / 创作 / PM / 设计师。他们不会自己做 context 管理，因此本能力对他们更应「无感」：不需要他们理解「为什么要 fork」这个概念，事情顺、该分开的自动分开、该带的上下文自动在那。

画像 真切换场景（应拦截） 对用户的价值
研发（理念原型） 在一个 session 里开发需求 A，中途撞见无关 bug B，顺手在同会话问 B。 避免 A 的会话被 B 污染、事后还得重开会话重新带上下文。
产品经理 正在写 A 需求的 PRD，突然「对了，帮我算下另一个功能的 ROI」。 PRD 的上下文不被打断，ROI 在独立 session 里更专注。
设计师 正在调 A 页面的视觉，突然「顺便给我出一版完全不同产品的 logo」。 A 的视觉语境不被另一产品的需求稀释。
三个场景的共同指纹完全一致：对象变了 + 完成它不会推进主线 + 它有独立的生命周期。判据可在研发 / 非研发间平移，说明这不是研发专属能力。

05核心概念与判定标准

判别器要对齐的靶子，是精准区分「会造成不可逆污染的真切换」与「主线内的自然漂移」。这是整个能力的生死线。

真切换 · 应当拦截

会造成不可逆污染

  • 对象变了：不是 A 的子步骤、不是对 A 的追问。
  • 做完不推进 A：解决它之后，A 的进度一格没动。
  • 有独立生命周期：它自己就能成为一个完整 session，不依赖 A 的上下文。
  • 触发是「撞见」：常由「突然发现」「对了顺便」「另外有个问题」引出。
自然漂移 · 不应拦截

是主线的自然推进

  • 追问 A 的某个细节（「这个边界 case 要不要写进 PRD」）。
  • 确认 A 的一个判断（「那这里应该怎么处理」）。
  • A 内部的一次重试或纠错。
  • 共同点：做完都让 A 前进，且离开当前上下文就不成立。
最硬的单条判据：自然漂移做完会让主线前进，真切换做完主线原地不动。判别器宁可漏报（放过一次可疑的真切换），不可在自然漂移上误报到产生高频打断。

06能力架构

完整框架包含六个模块：Session Task Brief、Intent Boundary Classifier、Boundary Policy Engine、Action Orchestrator、Handoff Generator、Artifact Ledger。它们把用户输入从「直接追加到当前聊天」变成「先在发送前理解边界，再选择合适的工作流动作」。

07功能需求

下列需求按模块编号（FR-x），每条给出关键需求点与验收标准（AC）。优先级 P0 第一阶段必做、P1 第二阶段、P2 后续。

FR-1 Session Task Brief（任务简报） P0

它到底是什么：一份后台维护的、结构化的「任务快照」。不是聊天记录的摘要，也不是给用户看的状态标签，而是判别器（FR-2）和 handoff（FR-5）唯一依赖的事实来源——回答「这个 session 此刻在做什么、已经定了什么、产出到哪了、什么不要再碰」。每个 session 一份，存为本地文件。下面是一个填好的实例，字段即设计：

{
  "session_id": "s_8f2a",
  "updated_at": "2026-06-30T10:12:00Z",

  // 主线：一句话，当前 session 在解决什么
  "main_task": "为「批量导出」功能写 PRD",

  // 最近推进：最近几轮实际推进了什么（记录事实，不判断"处于什么阶段"）
  "recent_progress": [
    "确定导出范围 = 当前筛选结果",
    "列出 3 个待定的字段格式问题"
  ],

  // 已确认判断：用户明确认可 / 否定 / 纠正过的（避免重开已结束的讨论）
  "confirmed_decisions": [
    "必须支持 CSV 与 Excel 两种格式",
    "不做定时导出（已明确砍掉）"
  ],

  // 已有成果：本 session 产出 / 引用的对象（详见 FR-6 Artifact Ledger）
  "artifacts": [
    { "type": "plan", "ref": "export-prd.md", "summary": "PRD 草稿 v0.1" }
  ],

  // 用户留下的问题：明确提出但未解决（继续追问 = 自然推进，不该被拦）
  "open_questions": [
    "导出大文件是否需要异步任务"
  ],

  // 明确排除：用户说不要的路线（避免被后续模型重新带回来）
  "excluded_directions": [
    "不导出敏感字段"
  ]
}
  • 更新时机：每轮对话后由一次轻量模型调用增量更新相关字段，而非每轮重写全文。
  • 存储：每个 session 本地目录一份文件为事实来源；运行时内存缓存，跨重启 / fork / handoff / 调试不丢。
  • 与 Artifact Ledger 的关系：Task Brief 是紧凑的工作快照，artifacts / confirmed_decisions 字段引用 FR-6 两层账本，不在此重复存全量。
验收标准
  • session 重启后能从本地文件完整恢复 Task Brief（六字段齐全）。
  • fork / handoff 时能读取该文件作为输入，不依赖完整聊天历史。
  • 可解释性：能据此说明系统为何建议 fork / compact。

FR-2 Intent Boundary Classifier（边界判别器） P0

它到底是什么：在用户输入进入主 agent 之前的一次专属模型调用。它不替用户做决定，只回答一个问题——「这句新输入和当前主线是什么关系，继续下去会不会造成不可逆污染」——并把判断输出成结构化结果，交给策略引擎（FR-3）。必须用模型，不得用关键词 / 相似度替代。

输入（从 Task Brief + 最近上下文组装）：main_task、recent_progress、confirmed_decisions、excluded_directions（均来自 FR-1）+ 最近 N 轮对话 + 用户刚输入但尚未发送的 pending 消息。

输出（结构化，把 §05 三条判据逐条显式化）：

{
  // 关系判定
  "relation": "same_task | sub_task | adjacent_task | new_task",

  // §05 三信号，逐条给出 —— 判别器的核心
  "signals": {
    "object_changed": true,        // 对象变了吗
    "advances_main_task": false,   // 做完会推进主线吗（最硬的一条）
    "independent_lifecycle": true  // 能独立成一个 session 吗
  },

  "pollution_risk": "none | low | medium | high",
  "confidence": 0.0,               // 0~1，低置信度直接归为 continue（不打扰）

  // 翻译成的建议动作（最终由 Policy Engine 裁决）
  "recommended_action":
    "continue | compact_context | fork_with_handoff",

  // 若需提示，给用户看的一句话（克制、可拒绝）；continue 时为空
  "user_message": "这个问题看起来会开启一个新方向，要我放到新 session 并带上上下文吗？"
}

判定映射（signals → 动作的基本规则，细化由 FR-3 承担）：

relationadvances_main_task典型 risk动作
same_task / sub_tasktruenone / lowcontinue
same_task（上下文压力高）true—compact_context
adjacent_task / new_taskfalsehighfork_with_handoff
任意（confidence 低）——continue（不打扰）

两个对照样例（同一主线 main_task =「为批量导出写 PRD」，看判别器如何分开真切换与自然漂移）：

pending 新输入relationadvancesrisk动作
「顺便帮我看下登录页转化率为什么降了」 new_task false high fork_with_handoff（对象变了、做完不推进 PRD、能独立成 session）
「导出要不要同时支持 CSV 和 Excel」 sub_task true none continue（追问主线细节、做完推进 PRD、离开上下文不成立）
验收标准
  • 在标注样本集上，自然漂移的误拦率（false positive）低于约定阈值（见 §11 / §13）。
  • 输出严格符合上述 schema；模型失败时兜底为 continue（不打断）。
  • 判别在发送前完成，延迟在可接受范围内（见 §13 延迟预算）。
  • 两个对照样例必须分别判为 fork_with_handoff 与 continue，作为回归基线。

FR-3 Boundary Policy Engine（策略引擎） P1

把判别器的语义结果翻译成产品动作，并施加阈值、用户偏好、风险等级与干预频率控制。

  • 低置信度不提示；同类场景被拒绝后降低频率。
  • brainstorming / exploratory session 默认更宽松。
  • 风险分级：低风险 continue、中风险 compact、高风险建议 fork。
验收标准
  • 用户「本次不再提醒」「仍在当前 session 发送」等选择能反馈进后续频率控制。
  • 同一 session 内不出现高频重复提示。

FR-4 Action Orchestrator（动作编排） P0

根据策略结果执行具体动作。底层动作复用已有能力，本模块只做编排。

动作场景前台行为底层动作
continue同一任务，边界清晰。不打扰。正常发送到当前 session。
compact_context同一任务，但上下文压力高、历史噪声重。默认后台整理，必要时轻提示。调用已有 Compact，继续当前 session。
fork_with_handoff真切换，继续会污染主线。展示克制建议，用户确认后创建新 session。生成 handoff → 创建 session → 发送新问题。
验收标准
  • compact_context 默认后台执行，不退化成新弹窗。
  • fork_with_handoff 在用户确认前不创建任何 session。

FR-5 Handoff Generator（交接生成） P0

fork 能否被接受，关键在 fork 后是否省事。Handoff 必须像一个高质量的任务启动包，明确带什么、不带什么。

FR-6 Artifact Ledger（产物账本） P1

区分「约束」与「产物」：约束是用户确认过、后续判断必须遵守的边界；产物是 session 过程中形成或引用的对象。边界判断需要同时知道二者。

  • accepted_constraints：用户确认过的原则、范围、禁区。
  • decisions：已定下的产品 / 技术判断。
  • artifacts：plans、files、reports、test_results、downloads。
  • unverified_claims：未验证但可能影响后续判断的信息。
验收标准
  • 约束与产物分两层存储，可分别被判别器与 handoff 引用。
  • handoff 生成时能据 ledger 决定「带哪些产物 / 守哪些约束」。

08交互与体验需求

前台体验极度克制。多数情况下系统在后台维护状态；只有当 fork 会明显避免不可逆污染时，才在发送前用一次轻确认介入。判据：每次介入应让用户「欸，它怎么知道」（惊喜），而非「它又来问我了」（负担）。

推荐文案

岔路口的轻确认

这个问题看起来会开启一个新的工作方向。为了保留当前 session 的主线，我可以把它放到一个新 session，并自动带上必要上下文。

  • Fork 并继续（自动带 handoff）
  • 仍在当前 session 发送
  • 本次不再提醒
避免文案

不要审判用户

避免「你偏离了主题」「这不是当前会话内容」「请开启新 session」这类表达。用户不是被纠正，而是被帮助组织工作流。

用户发送新输入

输入不立刻进入主 agent，而是先进入发送前的边界判断。

判别器读取 Session Intent Context

读取主目标、进展摘要、产物、健康信号和最近消息。

Policy Engine 选择动作

低风险继续，中风险 compact，高风险建议 fork。

Action Orchestrator 执行

执行 continue / compact_context / fork_with_handoff。

09技术方案与现有准备度

系统已有很多原料，但尚未形成稳定的 Session Task Brief 与 Artifact Ledger。结论：模型版能力可建设，但应先补齐任务简报与产物账本，而不是直接上弹窗。

能力输入 准备度 当前情况 需要补齐
当前主目标 部分具备 已有模型生成的 session title。 稳定的「主线目标」记录，不能只依赖标题。
最近进展摘要 部分具备 已有 conversation summary，用于 transfer / branch fallback。 每轮维护 rolling progress summary。
关键约束与产物 原料具备 约束散落在对话与 notes；产物散落在 plans、attachments、downloads、tool events。 拆成 Accepted Constraints 与 Artifact Ledger 两层。
最近几轮用户输入 已具备 消息完整保存在 session JSONL。 抽取窗口策略。
用户本次新输入 已具备 sendMessage 入口天然可获得。 在主 agent 执行前进入边界判断流程。
session 健康状态 大部分具备 有 messageCount、tokenUsage、contextWindow、status、labels。 coherence、phase churn、turn drift 等派生指标。
模型二级调用 已具备 已有 queryLlm / call_llm，支持 systemPrompt 与结构化输出。 产品专属 prompt、schema、失败兜底。

10分阶段路线图

不做所谓 MVP 弹窗。第一阶段也是完整框架中的最小闭环：先建立 Task Brief 与判别能力，再接入已有 Compact / Branch / Session 创建。

阶段范围包含需求出口标准
P0 · 最小闭环 后台维护 Task Brief；发送前判别真切换；确认后 fork 并自动 handoff。 FR-1 / FR-2 / FR-4（continue + fork_with_handoff）/ FR-5 真切换可在发送前被拦下；fork 后用户无需手动补上下文；自然漂移不被高频打断。
P1 · 策略与产物 引入策略引擎与产物账本；接入 compact_context（后台自动整理）。 FR-3 / FR-6 / FR-4（compact_context） 频率控制生效；handoff 能据 ledger 精准取舍；误拦率达标。
P2 · 增强 派生健康指标、企业可审计判断日志、跨 session 关联增强。 派生指标 / 审计 / 关联 企业模式保留可审计日志；多线管理体验联动。
第一阶段的不可妥协项：判断时机必须在发送前，且默认极度保守（宁可漏报，不可误报到高频打断）。先用「省事」赢得信任，再逐步放开更主动的介入。

11成功指标

Success Metrics

衡量是否真有价值

  • fork 建议接受率
  • 用户选择「仍在当前 session」的比例
  • fork 后新 session 完成率
  • 长 session 中重复解释 / 上下文修复回合减少
  • context pressure 下降
  • 用户关闭提醒比例（越低越好）
  • 边界判断 false positive 反馈量
关键护栏指标

准确率是生死线

  • 自然漂移误拦率（false positive）必须低于约定阈值。
  • 真切换漏拦率（false negative）单独追踪——它对应「整条 session 报废」的损失。
  • 单 session 内介入频次上限，防止打扰累积。
  • 「本次不再提醒 / 永久关闭」触发率作为信任流失预警。

12风险与对策

风险影响对策
判别器误拦自然漂移 高频打断 → 用户关闭功能 → 功能死亡且不再开启。 默认极度保守、低置信度不提示、被拒后降频、exploratory session 更宽松。
事后介入无法阻止污染 损坏已发生，弹窗只剩打扰。 判断时机锁定在发送前，作为第一阶段不可妥协项。
痛点「可忍」，用户不主动呼唤 价值难被感知，易被当作多余打扰。 以「无感增益」呈现（自动 handoff），让能力先以省事形态出现。
被误当作变现卖点 背上无法兑现的商业指标。 定位为护城河拼图 / 原生感兑现，不承诺独立付费转化。
具体切口仍是创始人体感 可能在做一个用户尚未感知的问题。 用 §11 护栏指标与下一轮用户研究证伪「分流」切口（见开放问题）。

13开放问题

  • 误拦率阈值定多少？ FR-2 / §11 的 false positive 上限需要一个具体数值，建议先用内测标注样本集校准。
  • 「分流」切口如何证伪？ 当前主要靠创始人体感，需在下一轮用户研究中专门验证非研发用户是否真的感知到「会话被污染、只能弃用」。
  • 第一阶段是否完全不主动弹窗？ 已确认时机必须在发送前；待定的是 P0 是否先只做「用户自己 fork 时自动 handoff」的静默增益，还是直接带上发送前轻确认。
  • 判别延迟预算？ 发送前的模型调用会引入延迟，需确定可接受上限与降级策略。
  • 企业模式审计范围？ 哪些判断日志需要可审计、保留多久。
  • Task Brief 与已有 summary 的关系？ 是复用 conversation summary 扩展，还是独立维护，需在技术设计阶段定。
